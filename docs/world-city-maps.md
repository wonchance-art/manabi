# 월드 계층형 맵 설계 — 전국맵 + 도시 정밀맵 (2026-07-13)

> 오너 지시: "지금 맵은 전국을 이동하는 맵으로 사용하고, 핵심 도시들은 더 큰 맵에서
> 동네 하나하나 자세하게 구현하며 돌아다닐 수 있도록."
> 마스터플랜 축 A(월드×학습 융합)의 공간 확장 — 문화 도어의 무대(편의점·이자카야·돈키·
> 라멘집)가 도시맵 안의 실제 '가게'가 된다.

## 1. 구조

```
전국맵(기존 448×384, WorldScene 'plaza')
  └─ 도시 노드 A버튼 → 도시 정밀맵(CityScene 'city:<id>')
       └─ 동네 구역·가게·NPC·출입구  → 가장자리 출구로 전국맵 복귀
```

- **CityScene 은 1개, cityId 로 파라미터화.** 도시 정의는 데이터 파일
  `src/components/world/cities/<id>.js` — 타일 그리드(도시 전용 인코딩), 구역(동네) 라벨,
  가게/NPC 배치, 출입구 좌표, 도시 미니맵. **도시 추가 = 데이터 파일 1개.**
- scene 식별자: `city:fukuoka` (기존 'plaza'·'airport'와 공존 — 자유 문자열).

## 2. 기존 인프라와의 접합 (전부 재사용)

| 시스템 | 접합 방식 |
|---|---|
| 멀티 피어 | scene 필드 기반 씬별 분리(applyPeersToScene)가 이미 동작 — city 값만 흘리면 도시별 자동 분리. net.js 무수정 |
| 근접 음성 | emitPeerDistances 헬퍼(같은 씬 실거리·타 씬 Infinity)를 CityScene update에 연결 |
| 채팅·말풍선 | 전역 채널 그대로(월드 공용 수다) — NPC 이벤트는 로컬(기존 원칙) |
| 좌표 영속화 | world_positions.scene 에 city:* 저장 허용 — persistable 정책 확장 + 도시별 스폰 검증(해당 도시 보행 타일) |
| NPC 대화 | npcScripts/NpcDialog 그대로 — 배치만 도시맵 좌표로 |
| 스탬프 | nodeId 체계 유지(도시 내 가게도 WORLD_NODES 등록 — kind: 'shop', city: 'fukuoka') |

## 3. 파일럿 — 후쿠오카 (1호 도시)

- **선정 이유**: 부산→후쿠오카 페리 동선의 목적지 + NPC 2곳 기구현. 도쿄는 독해 트랙
  (민준의 여정)과 묶어 2호로.
- 규모: 96×72 타일. 동네 3구역 — ① 하카타항(도착 부두·페리 연결) ② 하카타 라멘 골목
  (기존 라멘집 NPC 이전 배치, nodeId 유지) ③ 텐진 상점가(돈키호테 면세 카운터 —
  ot-12 면세 도어의 무대, 향후 NPC 후보).
- 다자이후 신사는 **전국맵에 유지**(별도 근교 지역 — 도시맵에 우겨넣지 않음).
- 타일셋 확장: 도로·보도·횡단보도·건물 파사드(노렌·간판 프리팹)·수로(나카강)·공원.
  도트 감성 유지(기존 sprites.js 문법).
- 전국맵 후쿠오카 노드: A버튼 → "후쿠오카 시내로 들어가기" 프롬프트 → CityScene.
  도시맵 남쪽 부두 출구 → 전국맵 후쿠오카 노드 앞 복귀.

## 4. 후속 로드맵

1. 파일럿 검증(멀티·좌표·NPC 이전) → 2호 **도쿄**(독해 트랙 A-2 연동: 하네다·시나가와·
   시부야 구역, 민준 발자국) → 3호 오사카/교토(간사이 도어 연계).
2. 도시맵 가게 = 문화 도어 도어 오픈(가게 앞 A버튼 → 해당 챕터/NPC) — "배운 곳에서 쓴다"
   루프의 공간화.
3. 도시맵 제작 파이프라인: 파일럿은 코드 선언(구획 그리드+프리팹), 도시가 늘면
   간단 에디터/제너레이터 검토.

## 5. 비원칙 (하지 않는 것)
- 도시맵에서의 채팅 분리(전역 유지 — 분리는 수요 확인 후).
- ~~실측 지도 재현(게임적 압축 — 동네의 '감성'이 목적, 정확도가 아님).~~ **← §6에서 폐기.**
- 파일럿 단계의 실내(가게 내부) 맵 — 가게는 파사드+대화/도어로. 실내는 후속.

## 6. v2 표준 — 실지형·고정 축척·무분할 (2026-07-13 오너 확정)

> 오너 지시: "실제 지도로 확인할 수 있는 지형. 다녀온 사람은 추억하고, 갈 사람은 지리·관광지
> 위치를 익히는 용도. 화끈하게 넓게. 도트를 낮춘 건 바로 이 지형·시스템을 극도로 올리기 위함."
> **§5의 '실측 재현 금지'를 폐기하고, 실좌표 기반 지형으로 전환한다.**

### 6.1 고정 축척 — 1타일 = 20m (전 도시 공통·불변)
- 모든 도시가 같은 m/타일 → 후쿠오카의 한 블록과 시부야의 한 블록이 같은 타일 크기.
  **일관성 = 타일 수가 아니라 축척이 같은 것.** 맵 타일 수는 도시 실측 폭÷20m로 자연히 달라진다.
- 예: 나하 3km→150 · 후쿠오카(타워/돔까지) 7.5km→375 · 도쿄 코어 12km→600.

### 6.2 무분할 — 도시 하나 = 맵 하나
- 도시를 동네로 쪼개지 않는다. 실측 전 범위를 한 맵에. (천장·분할 개념 폐기.)
- 대형 맵 도보 지루함은 **맵 안 전철역 fast-travel**로 해결(실제 대도시 경험 = 걷다가 전철,
  노선명이 학습 소재).

### 6.3 렌더 — 청크 RenderTexture 베이킹 (크기 무제한)
- Phaser 타일맵(셀당 Tile 객체 1개)은 600²=36만 객체로 메모리 폭발 → 폐기.
- 지형을 큰 오프스크린 RenderTexture 청크(예 512px)로 굽고 **보이는 청크만** 카메라에 그린다.
  1000²+ 도 프레임 비용 일정. 캐릭터·NPC·프리팹은 기존 스프라이트 그대로 그 위에.

### 6.4 지형 데이터 계약 (Codex geo 엔진 ↔ Claude 렌더/통합)
Codex가 실좌표에서 산출하는 `src/components/world/cities/<city>.geo.js`:
```
export const <CITY>_GEO = {
  meta: { city, bbox:[minLon,minLat,maxLon,maxLat], grid:{w,h}, metersPerTile:20, projection:'webmercator' },
  terrain: <길이 w*h 인코딩>,   // 값 = 아래 표준 지형 코드
  pois:    [ { id, nameJa, yomi, lat, lon, tile:[x,y], kind, facade? } ],
  stations:[ { id, nameJa, yomi, lat, lon, tile:[x,y], line? } ],  // fast-travel 노드
};
```
**표준 지형 코드**(현 `cities/fukuoka.js` CITY_TILE 승격 — 렌더·충돌 공용):
`ROAD · SIDEWALK · CROSSWALK · PLAZA · PARK · BEACH · WATER(차단) · RIVER(차단) · BUILDING(차단) · ISLAND(차단·배경) · BRIDGE · DOCK`.
- Codex: 해안선·강·도로·공원을 실좌표로 래스터화, POI/역 tile좌표를 투영으로 산출, **공식 자료 대조**로 상대 위치 충실도 + 육지 BFS 연결성(다리 포함) 검증.
- Claude: geo.js를 import해 청크 렌더 + POI/역에 일본어 간판·desc·프리팹·전철 fast-travel 배선.

### 6.5 접합 (기존 유지 — 회귀 금지)
scene `city:<id>` 씬별 피어/음성, city:* 좌표 영속·재접속 직행 스폰, resetScenePeers,
cityRedirectScene, EXIT→전국맵 복귀, NPC nodeId·스탬프 연속. 전 도시 공통.
