# 기획 v3 — 살아있는 연재 (Living Serial)

> **한 줄 요약.** 잊혀가는 단어가 다음 화의 재료가 되고(망각 예보), 앱 어디서 만난 단어든 하나의 기억으로 쌓이며(하나의 기억), 다음 화의 전개를 학습자가 직접 쓴다(공동 작가). 복습 알림이 아니라 **"다음 화가 도착했어요 — 당신이 쓴 대로 이야기가 흘러갑니다"** 가 재방문 사유가 되는 앱.
>
> 근거: `docs/roadmap-2026-07.md`(13-에이전트 진단·적대검증)와 코드 실측. v2(`mossy-cooking-sparrow`)를 대체하지 않고 **그 위에** 짓는다 — v2는 이미 거의 다 구현돼 있다.

## 1. 현재 학습 흐름 — 실측 지도

```
[진입]  홈/학습 → "오늘 학습"(/study)          ← PR#62로 콜드스타트 연결 완료
[세션]  워밍업 2문항(어제 오답+어시스트)
        → 문단 읽기 (연재! prevArc/arcSummary로 같은 인물이 이어짐 — studyParagraph.js:81-89)
        → 확인 7문항 (rung별 choice/typing/listening, EWMA 다이얼)
        → 격일 산출 → 클로징(내일 예고 nextPreview + 프리페치 — StudySessionPage.jsx:255)
[보조]  /vocab SRS덱 · /review/grammar · /writing · /listen · 사전(단어만/뜻만) · /materials 뷰어
[동기]  스트릭(+freeze) · 성장 통계 · 연재 예고. XP는 철거(의도적).
```

v2 설계 헌법(모든 노출은 인출 기회 · 85% 목표 · 실패해도 세션은 나온다)은 코드에 지켜지고 있다. **v2는 성공했다. 문제는 v2가 풀지 않은 것에 있다.**

## 2. 문제의 본질 — 세 개의 단절

### 단절 ① 재방문에 '당김(pull)'이 없다  ~~푸시 인프라 0~~
- `public/sw.js`에 push 핸들러 없음(grep 0건). 알림·이메일·예보 — 앱 밖에서 부르는 채널이 전무.
- 스트릭은 "잃지 않으려는" 소극 장치. 연재 예고(nextPreview)는 **세션 안에서만** 보인다 — 어제 세션을 안 한 사람은 예고도 못 본다. 재방문이 가장 필요한 사람에게 재방문 장치가 안 닿는 구조.
- FSRS 앱의 생명은 '제때'인데 제때를 알 방법이 없다. (진단 사각지대 #2 — "XP 철거 후 무엇이 재방문을 유도하나")

### 단절 ② 기억이 표면마다 조각나 있다
- viewer는 surface로, listen은 base_form으로 저장(`ViewerPage.jsx:620` vs `ListenLabPage.jsx:609`) — 같은 단어가 다른 키로 흩어져 rung이 안 쌓일 수 있다(진단 렌즈 교차 불확실성).
- **사전(단어만/뜻만)은 기억계 밖이다** — 사용자가 사전에서 가리고 맞히는 것은 명백한 인출 연습인데 이벤트가 한 건도 안 남는다. 열심히 외워도 /study는 모른다.
- 산출(produce)은 `source='writing'`이라 vocab rung 파이프라인에 **배선조차 안 됨**(`skillRung.js:88-97`) — rung 사다리 상단(4)이 죽어 있다. 학습과학적으로 산출이 가장 강한 인출인데.

### 단절 ③ FSRS의 숨은 보석을 안 쓴다
- ts-fsrs는 **retrievability(지금 이 단어를 기억할 확률)** 를 계산할 수 있다. 이건 "왜 오늘 와야 하는가"의 정직하고 정량적인 근거인데 어디에도 표시하지 않는다. due 개수(이진)만 보여준다 — "12개 밀렸어요"는 죄책감이고, "오늘 밤 '約束'이 70% 아래로 떨어져요"는 예보다.

**세 단절의 공통 뿌리: 각 표면이 자기만의 세계다.** 연재 엔진·FSRS·4언어 콘텐츠라는 강한 자산이 서로를 모른다.

## 3. 설계 — 세 개의 축

### 축 A. 망각 예보 (Forgetting Forecast)
FSRS retrievability를 **날씨로 은유**해 노출한다. 죄책감 프레임 금지.

- `src/lib/forecast.js`(신규, 순수함수): 유저의 SRS 행에서 retrievability 계산 → `{ 오늘 70% 아래로 떨어지는 단어 수, 대표 단어 실명 3개, 3분이면 구할 수 있음 }`
- 홈·학습 페이지 위젯: "🌦 오늘 흐림 — '約束' 외 11개가 안개로 들어가요. 3분이면 붙잡아요." 탭 → /study.
- 세션 클로징에 "오늘 구한 단어 n개" — 예보와 수확이 한 쌍(손실 회피 + 성취).
- **잊혀가는 단어의 실명 지목**이 차별점이다. "복습하세요"는 무시당하지만 자기가 아는 단어의 이름은 무시하기 어렵다.
- 원칙: 예보는 하루 1회 상한. '실패/밀림' 어휘 금지 — 날씨는 누구의 잘못도 아니다.

### 축 B. 하나의 기억 (One Memory)
"네가 만난 모든 단어는 하나의 기억이다"를 제품 원칙으로.

- **item_key 규약 통일**(v2 플랜 M3의 승격): 모든 표면이 word_text 단일 규약으로 읽고 쓴다. viewer/listen 저장 경로 정규화.
- **사전을 기억계에 편입**: 단어만/뜻만 모드에서 행 탭 시 1탭 자가 채점("기억났어/몰랐어") → review_events 기록(`source:'dict', qtype:'self'`). **FSRS에는 쓰지 않는다** — 비예정 조기 복습의 stability 왜곡 방지(v2 헌법의 L2와 동일 원칙: *예정된 인출만 FSRS를 움직인다*). rung·예보·재료 선정에만 참고.
- **산출 배선 봉합**: produce 이벤트가 vocab rung에 잡히도록 배선(source 정규화 또는 deriveVocabRungs 확장). rung 0→4 사다리가 처음으로 완주 가능해진다.
- 효과: 연재 문단 재료 선정이 "앱 전체에서의 만남"을 반영 → **어제 유튜브에서 만난 단어가 오늘 이야기에 등장하는 '우연한 재회'**. encoding variability의 극대화이자 감동 포인트.

### 축 C. 공동 작가 (Co-Author)
죽은 산출 슬롯에 서사적 이유를 준다. 작문을 '따로 하는 것'에서 '연재의 일부'로.

- 격일 산출 프롬프트를 연재 문맥으로 교체: **"주인공이 다음에 어떻게 할까요? 한 문장으로 쓰세요."**
- 기존 루브릭(0–3) 채점 유지. **≥2점이면 사용자의 문장이 arcSummary에 병합**되어 내일 문단 생성 프롬프트(prevArc)에 반영 — 이야기가 실제로 사용자가 쓴 대로 흘러간다.
- 학습자는 독자가 아니라 공동 작가가 된다. 산출(가장 강한 인출)을 하는 이유가 '숙제'가 아니라 '전개 결정권'이 된다. 내일 재방문 사유가 하나 더 생긴다: *내 문장이 어떻게 이야기가 됐는지 보러 온다.*
- 기술적으로 싸다: prevArc 주입 지점이 이미 있다(`studyParagraph.js:87`). 안전장치 — 루브릭 0–1점이면 미주입(기존 연재 유지), 프롬프트에 "사용자 문장은 소재로만, 문단 규칙·검증 패스 불변" 명시, 기존 결정적 검증 패스 그대로 통과해야 채택.

### 세 축의 결합이 만드는 루프
```
망각 예보가 부른다 → 오늘 화에 잊혀가던 단어가 등장한다(하나의 기억)
→ 확인 문항으로 구한다 → 마지막에 다음 화를 내가 쓴다(공동 작가)
→ 내일: "당신이 쓴 대로 이야기가 도착했어요" + 새 예보 → 다시 부른다
```
알림 카피가 "복습하세요"가 아니라 **"다음 화 도착 — 어제 당신이 정한 대로 미나는 역으로 갔어요. 그리고 '約束'이 오늘 밤 안개로 들어가요."** 가 된다.

## 4. 구현 로드맵 — 4단계 (각 단계 = 독립 PR·검증 게이트)

| 단계 | 내용 | 핵심 파일 | effort | 위임 | 완료 기준 |
|---|---|---|:--:|:--:|---|
| **P1. 배선 봉합** (선행조건) | item_key 통일(M3) · produce→rung 배선 · 사전 자가채점 이벤트(FSRS 미반영) | vocabIO/ViewerPage/ListenLabPage · skillRung.js · ReferenceVocabPage | 3 | opus | rung 단위테스트에 produce 승급 케이스 · 사전 탭→이벤트 적재 확인 · 기존 36 테스트 무회귀 |
| **P2. 망각 예보** | forecast.js 순수함수 · 홈/학습 위젯 · 클로징 "오늘 구한 단어" | 신규 lib/forecast.js · LearnPage/HomePage · StudySessionPage | 2 | sonnet(계산)+opus(UI판단) | retrievability vitest · 4언어 위젯 · 죄책감 카피 0 |
| **P3. 공동 작가** | 산출 프롬프트 연재화 · ≥2점 문장 arcSummary 병합 · 미주입 폴백 | studyParagraph.js · StudySessionPage · writingPrompts | 2 | opus | 루브릭 0–1점 미주입 재현 테스트 · 검증 패스 불변 · 산출 이벤트가 rung에 반영(P1 의존) |
| **P4. 앱 밖 당김** | Web Push(sw.js push 핸들러+VAPID+구독 테이블+발송 cron) 또는 저비용 선행안(주간 이메일 다이제스트) | sw.js · 신규 마이그레이션 · edge function | 4 | opus+사용자 결정 | 구독/거부 폴백 · 하루 1회 상한 · 예보+다음화 카피 |

- P1이 전부의 선행조건(진단이 지적한 item_key 불확실성 해소 겸). P2·P3은 P1 뒤 병렬 가능.
- P4는 인프라 결정(VAPID 키·발송 주체)이 필요해 **사용자 확인 후** 착수. 그 전까지는 P2의 인앱 예보가 당김을 대신한다.
- 각 단계 vitest + build + check-content 0 + 프리뷰 실사용. 로드맵 §3 기존 항목(admin RLS, EN/JA 어휘 보강)과 독립 — 병행 가능.

## 6. 스스로에 대한 레드팀 — 알려진 반론과 응답
- **"푸시 피로로 이탈 가속"** → 하루 1회 상한 + 사용자 주 학습 시각(이벤트 시각 중앙값)에만. 예보가 '없는 날'(떨어지는 단어 0)은 침묵.
- **"사용자 문장 주입 = 콘텐츠 안전 구멍"** → 3중 방어: 루브릭 ≥2점만 · 프롬프트에 소재 한정 명시 · 기존 결정적 검증 패스 통과 못 하면 폐기·폴백.
- **"retrievability 노출이 오히려 압박"** → 숫자·퍼센트 직접 노출 금지, 날씨 은유만. "밀림/실패" 어휘 금지.
- **"사전 자가채점이 FSRS 오염"** → FSRS 미반영 원칙 고정(위 B). 이 원칙 한 줄이 무너지면 spacing 전체가 무너진다 — 코드 주석으로 못 박을 것.
- **"연재가 부담(매일 긴 이야기?)"** → 문단은 지금과 동일한 3~5문장. 바뀌는 건 재료 선정과 산출 프레임뿐. 세션 6~8분 예산 불변.
