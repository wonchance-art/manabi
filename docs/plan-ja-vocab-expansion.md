# 기획 — 일본어 어휘 대확장 (JLPT 표준 커버리지)

> **한 줄 요약.** 4개 언어 중 어휘가 가장 빈곤한 일본어(981개)를 JLPT 표준 스케일(~8,000)로 끌어올린다. ZH에서 검증된 §4.7 방법론을 쓰되, JA 고유 위험인 **요미(읽기) 환각**은 LLM이 아니라 **kuromoji가 결정적으로 생성·검증**하는 것이 이 기획의 핵심 차별점이다.

## 1. 현황과 목표 (실측)

| 급 | 현재 | JLPT 표준(Tanos 계열 추정) | 목표 |
|---|---:|---:|---:|
| N5 | 169 | ~720 | ~700 |
| N4 | 185 | ~670 | ~650 |
| N3 | 255 | ~1,800 | ~1,700 |
| N2 | 234 | ~1,830 | ~1,750 |
| N1 | 138 | ~3,470 | 1,500 → 전량(선택) |
| **계** | **981** | ~8,500 | **~6,300 → 8,000+** |

- JA는 문형(936, 전 언어 최고)·문법 골격은 탄탄한데 어휘만 최하위 — 문법을 태울 어휘가 없다.
- n1.js는 "심화·faux-friends 선별 약 140개"로 **의도된 큐레이션** — 지우지 않고 표준 레이어를 옆에 얹는다(파일 무수정 원칙).

## 2. JA 고유 위험과 해법 (ZH와 다른 점)

| 위험 | ZH에는 없던 이유 | 해법 |
|---|---|---|
| **요미 환각** — 문장 전체 kana 읽기(`ex.yomi`)가 스키마 필수. LLM이 쓰면 한자 문맥 독음(明日 あした/あす, 行く いく/ゆく)을 틀린다 | 병음은 성조부호 형식 검사로 충분했음 | **ex.yomi는 LLM이 쓰지 않는다.** 예문(ja)만 생성 → 서버의 kuromoji로 요미를 **결정적으로 파생**(기존 `jaTokenize` 스택 재사용) → 기존 하드 게이트 `check-furigana.mjs`(요미 정렬 검사, prebuild exit 1)로 전수 검증 |
| **표제어 요미** | — | 소스 목록에 kana 읽기가 포함됨(골격의 일부) — 생성 금지, 그대로 복사 |
| **활용** — 예문에서 동사·형용사가 활용됨(食べる→食べた) → '예문에 표제어 실재' 검증이 정확 매칭으로는 실패 | 중국어는 무활용 | 어간 매칭 휴리스틱(사전형 마지막 가나 제외, 최소 2자) — cloze 단계(P1-b)에서 이미 검증된 방식 재사용 |
| **요미 갈림 단어** — 方(かた/ほう)·一日·辛い(からい/つらい) 류 | 다음자 44개를 분리 처리했던 것과 동형 | P1에서 기계 분리(소스에 복수 독음 표기·1자 한자어) → **Opus 전담 정밀 생성** |
| **merge 부재** — JA index.js에는 mergeVocab이 없다(급=단일 파일 직결) | ZH·FR은 이미 있음 | `mergeJaVocab` 신설 — FR 패턴(손작성 테마 보존 + 보강 테마 뒤에 병합, `ja` 기준 급 내 dedup) |

## 3. 파이프라인 (§4.7 방법론 + JA 특화)

```
P0 소스 실측(Opus) → P1 결정적 준비(Fable 코드) → P2 대량 생성(Sonnet 함대)
                                                 ↘ 위험어휘 정밀(Opus)
→ P3 결정적 검증(코드: 스키마·중복·어간실재 + kuromoji 요미 파생 + check-furigana)
→ P4 표본 언어 감사(Opus, 급별 30표본) → P5 조립·mergeJaVocab 등록·빌드·PR(Fable)
```

- **P0**: 소스 후보 실측 — 1순위 Tanos JLPT 리스트(tanos.co.uk, CC-BY: kanji·kana·급 포함), 2순위 elzup/jlpt-word-list(집계본). 라이선스 파일·수량·kana 포함 여부·급간 자체 중복을 실제 접근으로 확인(ZH 조사 때처럼 접근 실패는 실패로 보고). 기존 981과의 순증분 실측.
- **P1**: 다운로드 → 정규화 → 기존·급간 중복 배제 → 청크 40개 단위 분할 → 요미 갈림·1자 한자어 분리. ko 뜻 생성을 돕는 영문 gloss를 소스에서 동봉.
- **P2**: 청크당 규칙 고정(골격 무변조·ko 뜻·예문 ja/ko만 생성, **yomi 필드는 빈칸으로 남김**). H6에서 검증된 자가검증 프롬프트 재사용.
- **P3**: ① 스키마·개수·골격 무변조·중복 0 ② 예문 어간 실재 ③ **kuromoji로 ex.yomi 일괄 파생**(스크립트) ④ `check-furigana.mjs` 전 파일 통과 ⑤ check-content 오류 0. 실패 항목은 목록화해 재생성 또는 Opus 타깃 패치.
- **P4**: 급별 무작위 30표본 — 예문 자연스러움·번역 정확도·한국어 화자 관점(한자어 동원어 뜻이 어색하지 않은지). 계통적 결함이면 해당 청크 재생성.
- **P5**: `n{X}_jlpt_a.js…` 파일 조립(200개/파일, H6 관행) + index.js에 mergeJaVocab 등록 + vitest/build + PR.

## 4. 웨이브 분할 (승인 게이트)

| 웨이브 | 내용 | 순증분(추정) | 게이트 |
|---|---|---:|---|
| **W1 기초 완성** | N5+N4 표준 전량 | ~1,050 | 완료 후 오너 인앱 검수 — 품질 확인 뒤 W2 |
| **W2 중급** | N3+N2 (빈도 우선) | ~3,100 | 2개 서브웨이브(N3→N2)로 분할 |
| **W3 상급** | N1 빈도 상위 1,500 | ~1,450 | 잔여 ~1,900은 W3 결과 보고 결정 |

- 한 웨이브 = 한 PR. W1 규모는 H6 보강(1,000개)과 동급이라 파이프라인 실증을 겸한다.
- 콘텐츠는 서버 한정(§4.1)이라 번들 무영향이나, 빌드 시간·레지스트리 크기는 웨이브마다 모니터.

## 5. 모델 활용 계획 (역할·규모·예산)

| 모델 | 역할 | 규모(웨이브당) |
|---|---|---|
| **Fable 5** | 이 설계·오케스트레이션·P1/P3 **결정적 스크립트 작성과 실행**·웨이브 게이트에서 diff/표본 최종 검수. 생성 노동 금지(운영 규약 §2) | 오케스트레이션 오버헤드만 |
| **Opus 4.8** | ① P0 소스 실측·mergeJaVocab 설계 확정(1기) ② **요미 갈림·1자 한자어 전담 정밀 생성**(1기, H6 다음자 전담과 동형) ③ P4 급별 표본 감사(1기) ④ 발견 결함 타깃 패치(필요 시) | 3~4기 |
| **Sonnet 5** | P2 대량 생성 함대 — 청크 40개×2/기, 자가검증 포함(H6에서 14기·1,000개·불량 0 실증) | W1 ~14기 · W2 ~40기(서브웨이브 분할) · W3 ~19기 |
| **코드(모델 아님)** | 요미 생성(kuromoji)·요미 정렬(check-furigana)·스키마/중복/어간 검증·조립 | — |

**예산 추정(실측 기반)**: H6 1,000개 = Sonnet ~0.9M 토큰. → W1 ≈ 1.0M(Sonnet)+Opus 3기, W2 ≈ 3.0M, W3 ≈ 1.4M. 전량 완주 시 Sonnet 합계 ≈ 5.5~6.5M 토큰. Fable는 웨이브당 검수·조립 지휘만.

**원칙 재확인**: 요미·병음류 발음 데이터와 골격(표기·급·품사)은 **모델이 만들지 않는다** — 소스 복사 또는 kuromoji 파생. 모델은 한국어 뜻과 예문(원문·번역)만 만든다. 모든 생성물은 결정적 검증을 통과해야 리포에 들어간다.

## 6. 스스로에 대한 레드팀

- **"kuromoji도 고유명사·희귀 독음을 틀린다"** → 예문 생성 규칙에 고유명사 회피(인명·지명 금지)를 명시해 kuromoji의 강한 구간(일반 어휘 문장)만 쓰게 한다. check-furigana 실패분은 어차피 하드 게이트에 걸려 재생성된다.
- **"Tanos 리스트는 비공식(2010년 이후 공식 목록 없음)"** → 사실. 그러나 커뮤니티 표준으로 20년간 검증됐고 CC-BY로 깨끗하다. P0에서 라이선스 원문 확인을 의무화하고, 파일 헤더에 출처·라이선스를 명기(§4.7 관행).
- **"급당 700→1,700개면 사전 UI가 무거워진다"** → ZH H5(2,085)가 이미 그 규모를 실증(품사별 테마 자동 분류로 탐색 가능). 문제 생기면 그때 가상 스크롤 — 선제 최적화 안 함.
- **"N1 전량(+3,470)은 과잉"** → 그래서 W3를 빈도 상위 1,500으로 자르고 잔여는 데이터(N1 사용자의 사전 사용량) 보고 결정하는 게이트를 뒀다.
- **"한국어 화자에게 한자어 동원어(学校=학교)는 낭비"** → 낭비가 아니라 자신감 재료다(n5.js 자체가 '한자어 맛보기' 테마를 운영 중). 다만 뜻 생성 시 동원어는 한국 한자음 병기(예: "학교(學校)")를 규칙화해 학습 가치를 높인다.
