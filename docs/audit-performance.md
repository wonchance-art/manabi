## 결론

1. 기존 24 MiB 계약식부터 **도쿄 40.11 MiB**, **코트다쥐르 24.55 MiB**로 2도시가 실패한다.
   실제 런타임 임시 배열까지 더한 하한은 각각 40.96 MiB, 27.18 MiB다.
2. 서울(2.492M cells)·부산(1.470M)·레만호(1.047M)는 적응형 factor 덕분에 계약식은
   통과한다. 다만 런타임 하한은 서울 21.67 MiB, 부산 21.74 MiB로 24 MiB의 90%를 넘어
   headroom이 작다. 레만호는 15.47 MiB로 여유가 있다.
3. 신규 프랑스 3도시는 리옹 9.82 MiB, 보르도 10.87 MiB, 스트라스부르 8.27 MiB로 안전하다.
4. 로딩의 주 병목은 도시 진입 때의 grid 복사가 아니라 **앱 부팅 때 26개 geo 모듈을 모두
   정적 import하여 JS 리터럴을 파싱하고 RLE를 즉시 디코드하는 구조**다. 26도시 cold 평가
   중앙값 합계는 데스크톱 Chrome에서도 269.5 ms였다.
5. 실제 warm 도시 진입은 중앙값 평균 5.08 ms, 최악 도쿄 7.90 ms였다. `buildGrid`는 상위
   도시에서도 0.2 ms 이하이고, 진입 지점의 9~16개 청크 베이킹이 2.8~4.3 ms를 차지했다.

## 측정 계약과 한계

### 모바일 메모리

저장소의 도쿄/서울 계열 24 MiB 테스트 문법을 26도시에 동일 적용했다.

```text
계약 피크 = terrain Uint8Array
          + railway mask Uint8Array
          + entry grid Uint8Array
          + 표시 미니맵 RGBA backing
          + source 크기 RGBA 버퍼 × 2
```

여기에 현재 `Minimap` 구현이 새로 만드는 두 번째 `buildGrid()` 배열과, factor가 2 이상일 때
`downsampleCityGrid()`가 만드는 `codes`·`priorities`를 더해 **런타임 하한**도 계산했다.
이는 Phaser 텍스처, 브라우저 엔진, React 객체, GC 지연을 제외한 값이므로 전체 프로세스 RSS가
아니며 실제 피크보다 작을 수 있다.

- 기준선: 24 MiB = 25,165,824 bytes
- FAIL: 24 MiB 초과
- HIGH: 런타임 하한이 기준선의 90% 이상
- WATCH: 75% 이상
- OK: 75% 미만

### 로딩 체감

- 환경: nvm Node `v22.23.1`, Chrome 150, Phaser Canvas, 320×288 backing
- cold geo: 새 iframe realm에서 도시 adapter + geo 모듈 import 시간을 도시별 29회 측정
  (HTTP/Vite 변환 warm-up 1회 제외). JS 파싱과 module-evaluation RLE decode의 합이다.
- warm entry: 공용 텍스처를 먼저 구운 뒤 `buildGrid`, scene ready, Camera `preRender`, 진입 지점
  청크 베이킹을 도시별 29회 측정했다.
- 순위는 백그라운드 스케줄링 이상치에 둔감한 중앙값을 사용했다.
- 물리 모바일 기기 CPU/RSS 실측은 아니다. 절대시간보다 도시 간 상대 순위와 코드 단계별
  기여도를 판단하는 감사다.

### 메모리 해석

- 현재 factor는 `cells > 1,000,000`이라는 불연속 기준이다. 그래서 0.895M인 도쿄는 factor 1로
  남아 40.96 MiB까지 치솟는 반면, 더 큰 서울은 factor 3으로 내려가 21.67 MiB에 머문다.
- 코트다쥐르는 factor 2여도 가로 1,571칸이라 표시 canvas가 커서 24 MiB를 넘는다.
- 기존 테스트식은 런타임 임시 배열을 누락한다. 특히 서울은 계약식 78.2%에서 런타임 하한
  90.3%로, 부산은 81.8%에서 90.6%로 올라간다.
- 신규 프랑스 3도시는 모두 46% 미만이며, 이번 웨이브가 메모리 회귀의 원인은 아니다.

### 병목 지점

1. **JS 파싱 + RLE 즉시 디코드가 1순위다.** 26개 `.geo.js`는 합계 27,828,565 bytes
   (26.54 MiB)이고, module evaluation에서 terrain·railway RLE를 바로 `Uint8Array`로 푼다.
   26도시 cold 중앙값 합계 269.5 ms는 warm 진입 중앙값 평균 5.08 ms보다 훨씬 크다.
2. **`buildGrid`는 병목이 아니다.** 가장 큰 서울도 중앙값 0.2 ms이며 나머지 상위권은
   0.1 ms 수준이다.
3. **청크 베이킹은 warm 진입의 주 비용이지만 상한이 작다.** 320×288 viewport와 pad 1에서
   진입 위치에 따라 9·12·16개 청크를 굽고 중앙값 2.8~4.3 ms를 쓴다.
4. `create()`의 첫 `refreshChunks(true)`는 Camera `preRender` 전이라 좌상단 1청크를 먼저 굽고,
   첫 프레임에 실제 진입 위치 9~16청크를 다시 굽는다. 1청크 낭비는 작지만 제거 가능하다.

## 개선 제안 — 구현 금지

### P0 — 미니맵 예산 위반 즉시 폐쇄

1. factor를 고정 cell threshold가 아니라 **실제 byte 식을 만족하는 최소 factor**로 계산한다.
2. 도쿄는 factor 2로 올리면 계약식 약 11.95 MiB, 런타임 하한 약 13.23 MiB가 된다.
3. 코트다쥐르는 factor 3으로 올리면 계약식 약 13.83 MiB, 런타임 하한 약 15.97 MiB가 된다.
4. 일부 도시 테스트가 아니라 `CITY_MAPS` 26개 전수에 계약식과 런타임 하한 게이트를 건다.
   목표 headroom(예: 10% 또는 20%)은 오너가 확정해야 한다.

### P0 — 미니맵의 두 번째 grid 복사 제거

도시 씬이 활성 상태면 `Minimap`이 `city.buildGrid()`를 다시 호출하지 말고
`sceneRef.current.grid`를 읽도록 계약을 바꾼다. 도시별 0.10~2.38 MiB의 transient 배열을 없애고,
scene grid와 미니맵 지형의 단일 진실원도 강화할 수 있다.

### P1 — 무거운 geo를 진입 시점까지 지연

1. 현재 `cities/index.js`의 메타데이터 레지스트리와 무거운 geo loader를 분리한다.
2. `GameCanvas` 부팅 때 26개 scene class를 모두 만들지 말고 선택 도시를 dynamic import하여
   등록한다. 관리자 전체맵은 이름·크기·국가 같은 경량 manifest만 먼저 읽는다.
3. 이 변경의 목표는 초기 JS 파싱 대상 26.54 MiB와 cold 중앙값 합계 269.5 ms를 첫 화면에서
   제거하는 것이다. 도시 진입에는 해당 도시 1개의 decode 비용만 이동한다.

### P1 — RLE의 JS 리터럴 비용 축소

RLE 숫자쌍을 거대한 JS 배열로 파싱하지 말고 버전·길이·checksum을 가진 compact binary 또는
packed asset으로 분리한다. dynamic import만 적용해도 범위는 줄지만, 선택 도시의 JS parser 비용은
남는다. binary decode는 source parse와 terrain decode를 별도 performance mark로 측정할 수 있어
회귀 원인도 명확해진다.

### P2 — 첫 청크 순서와 계측 정리

1. 플레이어 생성·camera follow 뒤 최초 `refreshChunks`를 실행해 좌상단 1청크 선베이크를 없앤다.
2. `geo-import`, `build-grid`, `initial-chunks`, `scene-ready` performance mark를 개발 모드에 두고
   동일한 320×288 조건의 중앙값 회귀를 기록한다.
3. 청크 수 자체는 9~16으로 정상 상한이므로, geo lazy-loading과 메모리 P0보다 우선하지 않는다.
