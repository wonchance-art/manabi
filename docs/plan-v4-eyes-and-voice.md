# 기획 v4 — 눈과 목소리 (Eyes & Voice)

> **한 줄 요약.** v3가 루프의 엔진(예보→이야기→공동작가)을 만들었다. v4는 그 루프에 **눈**(무엇이 통하는지 측정하는 증거 회로)과 **목소리**(앱 밖에서 부르는 푸시)를 단다. 눈 없이 목소리를 달면 소음이고, 목소리 없이 눈만 있으면 루프가 앱 안에 갇힌다 — 둘은 한 사이클이다.
>
> 전제: v3 P1~P3 병합 완료(#62). 근거: docs/plan-v3-living-serial.md · docs/roadmap-2026-07.md(사각지대 #2 리마인더, #4 PII, #7 퍼널 계측) · 코드 실측(vercel.json cron 2건 가동 중, admin/metrics 존재하나 RLS로 본인 행만, push/email 스택 0).

---

## 1. 두 개의 결핍

### 결핍 ① 눈 — 우리는 지금도 장님이다
- v3 기획 §5가 지표 5종(D1 재방문·예보 탭률·공동작가 제출률·연재 연속일·재회 빈도)을 **정의**했지만, **측정할 방법이 없다**: `admin/metrics`는 RLS 때문에 관리자 본인 행만 집계한다(진단 렌즈가 impact 5로 지목).
- 13-에이전트 진단조차 스스로 인정했다 — *"어느 화면이 사용자를 잃는지 실측 없이 코드 구조로만 추론"*. v3의 성패 판단, P4 푸시의 카피·시각 선택, 다음 콘텐츠 투자 — 전부 증거 없이는 감이다.
- 단순 "admin 전체 SELECT 정책"은 함정이 있다: `review_events.detail`에는 오답 프롬프트·사용자 응답, `study_paragraphs`에는 사용자 작문이 들어 있다(진단 사각지대 #4 PII). **행 단위 admin 조회를 여는 것은 필요 이상의 권한이다.**

### 결핍 ② 목소리 — 안 오는 사람에게 닿을 방법이 없다
- v3 예보 위젯의 근본 모순: **앱을 열어야만 보인다.** 재방문 장치가 재방문한 사람에게만 작동한다.
- FSRS의 약속은 '제때 복습'인데, 제때를 알릴 채널이 0이다(sw.js push 핸들러 없음, 이메일 스택 없음).
- 자산은 이미 있다: PWA(sw.js+manifest) · Vercel Cron 가동 선례 2건(`fetch-suggestions`, `backfill-ipa`) · 예보 계산(forecast.js) · 연재 예고(nextPreview). **부족한 건 배관뿐이다.**

## 2. 설계 원칙 (v4 헌법)
1. **집계만 눈에 보인다** — 관리자든 대시보드든 행이 아니라 숫자만. 자유 텍스트(작문·응답)는 어떤 화면에도 흐르지 않는다.
2. **침묵이 기본값** — 보낼 가치(떨어지는 단어>0 또는 새 화 도착)가 없는 날은 보내지 않는다. 하루 1회 상한. "밀림/해야" 어휘 금지(v3 헌법 승계).
3. **가치 먼저, 권한은 나중** — 설치 직후 권한 팝업 금지. 첫 세션 클로징(내일 예고를 본 직후)에만 묻는다.
4. **측정 없는 발송 없음** — 푸시는 눈(V4-1)이 켜진 뒤에 쏜다. 전후 D1 비교가 가능해야 한다.

## 3. Part 1 — 눈: 증거 회로 (Evidence Loop)

### 3.1 집계 전용 RPC (PII 해법과 RLS 해법을 한 번에)
- admin 전체-행 SELECT 정책을 **열지 않는다.** 대신 `SECURITY DEFINER` RPC 함수 4~6개가 **집계 결과만** 반환:
  - `admin_funnel(days)` — 가입→첫 세션→D1 재방문→D7 유지 (user_id는 카운트로만 소멸)
  - `admin_daily_metrics(days)` — 일별 세션 수·완주율·평균 문항 정답률(EWMA 원료)
  - `admin_v3_metrics(days)` — 예보 탭률·공동작가 제출률(산출 노출 대비)·연재 연속일 분포·재회 빈도
  - `admin_content_health()` — 문항별 오답률 상위(불량 문항 탐지, item_key만·detail 미노출)
- 함수 안에서 `is_admin` 검사 후 집계 — 함수가 곧 권한 경계. 행 반환 함수 금지를 마이그레이션 주석에 못 박는다.
- **마이그레이션 1건**(함수들 + 필요시 인덱스). 기존 테이블 스키마 불변.

### 3.2 UI 이벤트 규약 (마이그레이션 0)
- 예보 탭·푸시 열림 같은 **행동 계측**은 기존 `review_events`에 `source:'ui'` 규약으로: `{source:'ui', qtype:'forecast_tap'|'push_open'|'push_optin', item_key:'-', detail:{...}}`.
- 저용량(하루 수 건)·학습 이벤트와 명확히 구분(source로 필터)·rung/FSRS 계산은 source:'vocab'만 보므로 오염 없음(P1-b에서 검증된 필터).
- 규약을 `docs/architecture-and-handoff.md`에 1절 추가 — 이벤트 규약의 단일 출처 유지.

### 3.3 admin 대시보드 v4
- 기존 `admin/metrics` 페이지를 RPC 호출로 전환 + 퍼널 카드·v3 지표 카드 추가. 차트 라이브러리 추가 금지(기존 표현 문법 유지).

## 4. Part 2 — 목소리: 앱 밖 당김 (Web Push)

### 4.1 구독 배관
- **`push_subscriptions` 테이블**(v4 유일한 신규 데이터 테이블): user_id·endpoint·keys jsonb·lang·preferred_hour(smallint, 로컬 발송 시각)·created_at. RLS own-row 4정책. UNIQUE(endpoint).
- `public/sw.js`에 `push`·`notificationclick` 핸들러(클릭 → `/study`). VAPID 방식(무료, 서버 키 1쌍).
- 구독 UX — 원칙 3: 첫 세션 클로징의 내일 예고 아래 「내일 이야기가 도착하면 알려드릴까요?」 1버튼. 거부 시 재요청 금지(30일 침묵), 이후 학습 탭 설정에서만 수동 켜기.
- **iOS 한계 정직 고지**: iOS 16.4+에서 홈 화면 추가(PWA 설치) 후에만 푸시 가능 — 안내 문구 1줄, 억지 유도 금지.

### 4.2 발송 회로
- **발송자 = Vercel Cron**(선례 재사용): `/api/cron/send-forecast`를 **매시 실행**, `preferred_hour == 현재 UTC시`인 구독자만 발송. preferred_hour 기본값 = 가입 후 관측된 세션 시각 중앙값(없으면 20시 로컬 근사).
- **카피 엔진** — 예보 × 연재 결합(서버에서 forecast.js·study_paragraphs 프리페치 행 재사용):
  - 둘 다 있으면: 「미나가 역으로 갔어요 — 다음 화 도착. 그리고 '約束'이 오늘 밤 안개로 들어가요.」
  - 예보만: 「'約束' 외 3개가 오늘 밤 안개로 들어가요. 3분이면 붙잡아요.」
  - 새 화만: 「당신이 정한 전개로 다음 화가 도착했어요.」
  - **둘 다 없으면 발송하지 않는다**(원칙 2). 제목·본문 언어는 한국어(UI 언어), 단어만 학습언어.
- 발송 기록: `review_events source:'ui', qtype:'push_sent'` (상한 검사 겸 측정 — 3.2 규약 재사용). 실패(만료 endpoint 410)는 구독 행 삭제.

### 4.3 명시적 스코프 밖
- 이메일 다이제스트 — 스택 부재(신규 벤더 계약 필요), 푸시 성과 측정 후 재론.
- 네이티브 앱 푸시 — PWA로 충분한지 데이터부터.

## 7. 레드팀 — 반론과 응답
- **"집계 RPC도 결국 우회로 뚫린다"** → 함수 시그니처가 숫자만 반환 + `is_admin` 함수 내부 검사 + 행 반환 함수 금지 원칙을 마이그레이션 주석·핸드오프 문서에 이중 명기. 완벽이 아니라 최소권한이 목표.
- **"푸시 피로 → 앱 자체 이탈"** → 침묵 기본값·하루 1회·해지율 경보 지표·V4-4 판정 게이트(성과 없으면 끈다는 결정을 기획에 내장).
- **"preferred_hour 추정이 틀리면?"** → 기본 20시 로컬 근사 + 설정에서 수동 변경. 중앙값은 관측 5세션 이상일 때만.
- **"iOS에서 안 되는데 만들 가치?"** → Android/데스크톱 PWA는 즉시 작동. iOS는 설치 유도 문구로 점진 — 옵트인율 데이터가 iOS 투자 여부를 알려줄 것(눈이 먼저인 이유).
- **"review_events에 ui 이벤트 섞는 건 스키마 남용"** → 저용량·source 필터로 격리·마이그레이션 0의 실리. 월 1만 건 넘어가면 분리 테이블로 승격한다는 조건을 규약에 명시.
