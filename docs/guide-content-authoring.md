# 4트랙 콘텐츠 저작 가이드

이 문서는 일본어·프랑스어·영어·중국어의 문법·어휘 콘텐츠를 새로 쓰거나 초안을
정식 콘텐츠로 승격할 때 지켜야 할 공통 규약이다. 현재 스키마의 단일 기준은
`src/content/__tests__/contentSchemaContract.test.js`와
`docs/audit-content-schema.md`이며, 레슨으로 옮기는 규칙은
`docs/rfc-lesson-model.md`를 따른다.

## 1. 저작 단위와 파일 위치

콘텐츠는 `src/content/<track>/{grammar,vocab}/*.js`에 둔다.

- 문법 저작 단위: `default export` 챕터 배열의 챕터 1개
- 중국어 추가 예문: `*_examples.js`의 `챕터 slug → 섹션 index → 예문 배열`
- 어휘 원천 모듈: `vocab/*.js` 파일 1개
- 신규 어휘 세트: `themes[]`의 테마 1개, **8~10어**

기존 `a1.js`, `_flelex`, `_jlpt`, `_hsk` 계열은 여러 레슨이 공유하는 대형 원천
풀이다. 이 파일들은 호환성을 위해 모듈당 3어 하한만 유지한다. F3 이후 신규 확장 저작은
`*_draft_expansion.js`에서 시작해 정식화 때 `*_expansion.js`로 이름을 바꾼다.
이 이름은 린터가 신규 8~10어 세트 규칙을 계속 적용하는 신호다.

## 2. 문법 필드

모든 트랙의 문법 챕터는 다음 공통 필드를 갖는다.

| 용도 | 필드 | 규칙 |
|---|---|---|
| 식별 | `slug` | 비어 있지 않은 문자열, 트랙 안에서 영구 불변·유일 |
| 배치 | `level`, `order` | 레벨 문자열, 양의 정수 |
| 제목 | `title`, `topic`, `titleFr` | 비어 있지 않은 문자열. `titleFr`는 전 트랙이 쓰는 역사적 원어 부제 필드 |
| 안내 | `summary`, `duration` | 비어 있지 않은 문자열 |
| 본문 | `sections` | 1개 이상. 모든 섹션은 `heading`, 챕터 안 적어도 한 섹션은 `body` 필요 |

트랙별 예문 필드는 다음과 같다.

| 트랙 | 문법 예문 필수 필드 | 선택·특수 필드 |
|---|---|---|
| 일본어 | `ja`, `yomi`, `ko` | `note`, 챕터 `kana`, `kanjiExempt` |
| 프랑스어 | `fr`, `ko` | `ipa`, `note`, 활용표 |
| 영어 | `en`, `ko` | `ipa`, `note` |
| 중국어 | `zh`, `pinyin`, `ko` | `note`; 추가 예문은 `*_examples.js`에 같은 형식으로 작성 |

챕터마다 구체 예문을 **2개 이상** 둔다. 일본어 문자표처럼 문장 예문이 교육적으로
맞지 않는 경우에만 비어 있지 않은 `kana`와 `gojuon` 또는 `table`을 함께 두어 대체할
수 있다. 예문은 현재 챕터까지 배운 조각으로 이해할 수 있어야 하며, 상위 문법을
설명 없이 끌어오지 않는다.

## 3. 어휘 필드

어휘 모듈은 `{ level, title, desc, themes }`, `{ themes }`, 또는 기존 보강용 테마 배열을
내보낸다. 정식 shell에 `level`이 있으면 `title`과 `desc`도 비어 있지 않아야 한다.
모든 테마는 `name`, 비어 있지 않은 `words`, 선택 `icon`을 갖는다.

| 트랙 | 어휘 필수 필드 | 예문 `ex` 필드 | 선택 필드 |
|---|---|---|---|
| 일본어 | `ja`, `yomi`, `ko`, `pos`, `ex` | `ja`, `yomi`, `ko` | `hanja`, `id` |
| 프랑스어 | `fr`, `ipa`, `ko`, `pos` | `fr`, `ko` | `en`, `etym`, `gender`, `id` |
| 영어 | `en`, `ipa`, `ko`, `pos` | `en`, `ko` | `etym`, `id` |
| 중국어 | `zh`, `pinyin`, `ko`, 문자열 `pos`, `ex` | `zh`, `pinyin`, `ko` | `hanja`, `id` |

프랑스어·영어의 과거 보강 풀에는 `ex`가 없는 항목이 있어 현행 계약은 모듈당 예문
1개만 요구한다. **신규 8~10어 세트는 네 트랙 모두 모든 어휘에 예문을 쓴다.**
중국어 과거 HSK 3.0 항목은 빈 `pos` 문자열이 남아 있어 key와 문자열 타입까지만
호환한다. 새 항목의 `pos`는 반드시 실제 품사 값으로 채운다.

명시적 `id`를 쓰면 트랙 안에서 고유하고 영구 불변이어야 한다. `id`가 없는 기존
어휘는 레슨 매핑 때 `id || slugify(표제어)` 순서로 안정 키를 만든다. 표제어 중복은
학습상 필요할 수 있으므로 중복 ID와 같은 것으로 보지 않는다.

## 4. Course → Unit → Lesson 매핑

`docs/rfc-lesson-model.md`의 점진적 어댑터 원칙을 사용한다. 원본 콘텐츠를 복사하거나
DB에 먼저 옮기지 않는다.

| 원본 | 레슨 모델 매핑 |
|---|---|
| 트랙 × 레벨 | `Course`, ID는 `<track>-<level>` |
| 챕터 `order` 3개 묶음 | `Unit`, ID는 `<track>-<level>-u<ceil(order/3)>` |
| 문법 챕터 | `Lesson`, ID는 `<track>-<level>-<chapter.slug>` |
| `sections[]` | `grammar.chapterRefs[{ chapterSlug, sectionIndices }]` |
| 문법 개념 | `grammar.concepts`; 설명은 원문 참조 또는 짧은 해요체 요약 |
| 어휘 테마 | `vocabulary.themeRef`, `coreWords`, `newWordsCount` |
| 트랙 특수값 | `specialFields`; 3종을 넘으면 별도 RFC |

한 Lesson은 문법 개념 1개 이상, 8~10어 어휘 세트, 연습을 함께 묶어 15~20분을
목표로 한다. 원본 챕터가 25분 이상이면 자동으로 자르지 말고 저자가 두 레슨 분할을
검토한다. 같은 Unit 안의 이전·다음 연결은 `order`로 계산하고, Unit이나 Course를
넘는 선행 관계는 `prerequisites`에 수동으로 쓰고 이유를 저자 주석에 남긴다.

중국어 `*_examples.js`는 해당 `chapterSlug`와 섹션 index의 본문 예문에 병합한다.
프랑스어·영어의 기존 대형 어휘 풀은 훼손하지 않고 Unit 단위로 느슨하게 참조하되,
신규 레슨부터 8~10어 `*_expansion.js` 테마를 우선 연결한다.

## 5. DRAFT 규약

다음 중 하나면 초안이다.

1. 파일명에 대소문자 무관 `_draft`가 있음
2. JavaScript의 `//` 또는 `/* ... */` 주석 안에 대문자 토큰 `DRAFT`가 있음

콘텐츠 문자열의 소문자 `draft`는 초안 신호가 아니다. 초안에는 파일 머리에 목적,
담당 검수자, 소비 경로 미연결 상태를 적는다. 초안은 레지스트리·페이지·도어에
연결하지 않는다.

초안 작업 중에는 다음 명령으로 정식 콘텐츠와 함께 검사한다.

```sh
node scripts/lint-content.mjs --drafts
```

정식 승격 전에는 `--drafts` 결과의 error가 0이어야 한다. 그 뒤 콘텐츠 오너 검수를
받고 `_draft`와 주석의 `DRAFT` 표식을 모두 제거한다. 신규 어휘 파일은
`*_expansion.js` 이름을 유지해 8~10어 게이트가 승격 뒤에도 적용되게 한다.

## 6. 문체·품질·IP 검수

- 세트: 신규 어휘 테마당 8~10어, 모든 단어에 자연스러운 예문
- 문법: 챕터당 예문 2개 이상, 예문은 선행 학습 범위 안에서 작성
- 문체: 설명·요약·팁은 부드러운 해요체. 예문 번역은 원문의 반말·격식·문어체를
  보존할 수 있으므로 무조건 `요`로 고치지 않음
- 구조: `slug`와 명시적 `id` 중복 금지. URL·저장 키로 쓰인 뒤에는 rename 금지
- 레슨: 문법 개념 + 어휘 세트 + 연습, 15~20분. 25분 이상이면 수동 분할 검토
- IP: 실재 상호·인물·작품·브랜드·국기·엠블럼을 재현하지 않음. 일반화한 장소·상황과
  자체 예문을 사용하고, 정치 서술은 넣지 않음

해요체 자동 검사는 번역 문체를 훼손하지 않도록 결정적 표본만 본다. 문법 모듈은 첫
챕터 `summary`, 정식 shell 어휘 모듈은 `desc`가 `/요[.!?…]*$/`로 끝나는지 warning으로
출력한다. warning은 저자 검수 신호이며 exit code를 실패시키지 않는다. 최종 검수에서는
표본뿐 아니라 신규 설명 산문 전체를 읽는다.

## 7. 린터 실행과 판정

```sh
# 정식 콘텐츠만
node scripts/lint-content.mjs

# 정식 + DRAFT
node scripts/lint-content.mjs --drafts
```

출력은 `파일:줄:열 severity 규칙 메시지` 형식이며 파일·위치·규칙 순으로 고정된다.
종료 코드는 0(오류 없음), 1(콘텐츠 오류), 2(CLI 또는 내부 실행 실패)다. warning만
있으면 종료 코드는 0이다.

린터는 다음을 검사한다.

- #545 공통 9필드, 트랙별 문법·어휘·예문 필드와 shell
- 문법 예문 2개, 기존 어휘 모듈 3어 하한, 신규 세트 8~10어·전 항목 예문
- 트랙 안 챕터 `slug`와 명시적 `id` 중복
- 모듈별 해요체 대표 산문 표본
- 기본 실행의 DRAFT 제외와 `--drafts`의 DRAFT 포함

error가 나오면 콘텐츠 파일을 이 작업에 끼워 고치지 말고, `파일:위치` 목록을 콘텐츠
오너에게 넘겨 별도 정본화 작업으로 처리한다.
