# Q2 타일 정합 정정 삽입 지점 사전조사

## 0. 결론

**권장안은 (a) 생성 파이프라인 안에 버전·해시가 고정된 정정 패스를 넣고 geo를
재생성하는 방식이다.** 정정은 최종 타일 배열을 만드는 기존의 모든 변형 패스 뒤,
RLE 직렬화와 최종 통계·결정성 해시를 확정하기 전에 적용한다. 적용 후에는 지형을
더 고치는 자동 정규화를 다시 돌리지 않고, 4방 연결성·마커·BRIDGE·도시별 프로필을
읽기 전용으로 재검증해 위반 시 실패해야 한다.

직접 산출물 후처리인 (b)는 한 스크립트로 26도시를 훑기 쉬워 보이지만, generator를
다시 실행하면 정정이 사라지고 `Generated by ... Do not edit by hand`라는 provenance와
실제 산출 경로가 갈라진다. 여러 도시 테스트가 `build*Geo()` 결과와 커밋된
`*.geo.js`를 비교하므로 현재 결정성 계약과도 직접 충돌한다.

exact base `4bec11e9468cdbed363f59b2d39ccea38b2ebb43`에서
[`src/components/world/cities`](../src/components/world/cities)의 26개 `*.geo.js`를
전수 대조한 결과:

- 26/26이 `Generated by scripts/build-*.mjs` 헤더를 가진다.
- 26/26이 `scripts/data/<city>-osm-v*.json` 스냅샷을 가진다.
- 헤더가 가리키는 generator도 26/26 존재한다(21개 generator 진입점이 공용·전용으로
  26개 산출물을 담당).
- 따라서 **OSM 스냅샷이나 재생성 경로가 없는 비파이프라인 구세대 도시는 0개**다.
- `scripts/verify-city-geo.mjs`에는 22개 도시 프로필만 있다. 도쿄·오사카·교토·후쿠오카
  4개는 generator·스냅샷은 있지만 공용 verifier 프로필이 없고 도시별 geo Vitest가
  결정성·타일 질량·BFS를 고정한다.

본 문서는 Q1 스캐너가 산출할 고립 건물·부적절한 횡단보도·방향 어긋남·끊김 후보의
**적용 경로만** 설계한다. 아직 main에 없는 Q1 결과 좌표를 추정하거나 자동 수정하지
않으며, 제품 코드·geo·generator·공용 verifier·테스트·DB는 수정하지 않는다.

## 1. 리옹 생성 파이프라인

### 1.1 고정 입력

리옹 generator의 입력 계약은 다음과 같다.

| 항목 | 값 / 근거 |
|---|---|
| bbox | `[4.79, 45.71, 4.90, 45.80]` |
| 해상도 / 격자 | `20m/tile`, `428×501 = 214,428` cells |
| 투영 | Web Mercator + 중심 위도 cosine 보정 |
| 스냅샷 | `scripts/data/lyon-osm-v21.json` |
| 스냅샷 파일 SHA-256 | `f49e5b1ec1d28f65a47dea4e8b1018959c4546f77804f205cbf77827302cac53` |
| 원본 Overpass SHA-256 | `6f3511956d95e2cfd3e5de183a34110234027c3226336027b94be5f38f3e3147` |
| 현 geo 파일 SHA-256 | `0bb8a0a2983df4cfbaa2387fa8905610508ef5ecaa41d5d3992bf33ca41936d2` |
| 현 terrain SHA-256 | `1d3036f8c0d8347946e62f2e08d22152ff07ce6f33b05909d6db3d71335835c9` |
| 현 PNG SHA-256 | `2d1aac685c8a0bd82e98f210dd2239aeaee4e9acde1b6b71b103d03ab620759e` |

`buildLyonCityGeo()`는 스냅샷 텍스트의 SHA를 먼저 검사하고
([`build-lyon-city-geo.mjs` 414~449행](../scripts/build-lyon-city-geo.mjs#L414)),
고정 입력이 다르면 terrain 생성 전에 실패한다. 즉 Q2 정정은 OSM 스냅샷 자체를
고치는 작업이 아니라, 고정된 스냅샷에서 나온 최종 지형에 명시적 override를 적용하는
별도 버전 계층이어야 한다.

### 1.2 래스터화: raw OSM → 스냅샷 마스크

`lyon-osm-v21.json`은 raw OSM 요소가 아니라 이미 20m 격자로 래스터화된
스냅샷이다. 생성기는
[`build-french-city-osm-snapshot.mjs`](../scripts/build-french-city-osm-snapshot.mjs)가
담당한다.

1. raw 요소를 `type:id`로 정렬해 입력 순서 영향을 제거한다
   ([414~425행](../scripts/build-french-city-osm-snapshot.mjs#L414)).
2. 폴리곤은 scanline fill, 선형은 고정 반경 원판을 보간해 각각의 독립 마스크에
   그린다
   ([253~310행](../scripts/build-french-city-osm-snapshot.mjs#L253)).
3. 태그로 레이어를 분류한다.

   - `building=*` 폴리곤 → `building` mask 값 1.
   - 모든 `highway=*` way → `roadStyle()`의 반경과 값으로 `road` mask.
     자동차 도로 계열은 2, 보행로 계열은 1, `bridge!=no`는 3이다
     ([333~343행](../scripts/build-french-city-osm-snapshot.mjs#L333),
     [433~444행](../scripts/build-french-city-osm-snapshot.mjs#L433)).
   - `highway=crossing` node → 바닥 투영한 `[x,y]` 후보. 중복을 제거한 뒤
     `y,x` 순으로 정렬한다
     ([428~431행](../scripts/build-french-city-osm-snapshot.mjs#L428),
     [486~487행](../scripts/build-french-city-osm-snapshot.mjs#L486)).
   - 수면·하천·공원·산지·철도도 각각 별도 mask에 그린다.
4. 각 mask를 row-major RLE로 직렬화하고 RLE별 SHA-256을 스냅샷에 기록한다
   ([488~515행](../scripts/build-french-city-osm-snapshot.mjs#L488)).

리옹 스냅샷에는 `buildingRle`, `roadRle`, `waterRle`, `riverRle`, `parkRle`,
`mountainRle`, `railwayRle`와 횡단보도 후보 9,624개가 들어 있다. 이 단계에서
건물·도로는 서로 다른 mask이고 아직 최종 `CITY_TILE` 우선순위가 정해지지 않는다.

### 1.3 분류: 스냅샷 마스크 → source terrain

`buildTerrainFromSnapshot()`은 전 셀을 `SIDEWALK`로 시작한 뒤
`applySnapshotMasks()`를 호출한다
([`build-french-city-geo-core.mjs` 603~607행](../scripts/build-french-city-geo-core.mjs#L603)).
실제 덮어쓰기 순서는 다음과 같이 고정돼 있다
([382~417행](../scripts/build-french-city-geo-core.mjs#L382)).

1. `WATER`
2. `RIVER`
3. `MOUNTAIN` (수면 제외)
4. `PARK` (수면 제외)
5. `BUILDING` (수면 제외)
6. 도로 class

   - class 3이 수면 위에 있으면 `BRIDGE`
   - class 2 이상이 육지에 있으면 `ROAD`이며, 앞서 놓인 `BUILDING`도 덮는다
   - class 1은 건물이 아닌 셀만 `SIDEWALK`로 만든다

7. crossing 후보는 해당 셀의 `roadMask >= 2`이고 현재 terrain이 정확히 `ROAD`일 때만
   `CROSSWALK`로 바뀐다.

따라서 세 타일의 최초 결정 지점은 다음과 같다.

| 타일 | 최초 결정 | 이후 바뀔 수 있는 단계 |
|---|---|---|
| `BUILDING` | OSM `building=*` mask를 5번째로 적용 | class≥2 도로가 덮음; 리옹 소성분 collapse가 추가 건물 생성; 마커 anchor가 `PLAZA`로 제거 |
| `ROAD` | OSM `road` class≥2의 육지 셀 | 교량 정규화가 `BRIDGE→ROAD`; 입구 회랑·성분 연결 carving이 `ROAD` 추가; 소성분 collapse가 일부 `ROAD→BUILDING` |
| `CROSSWALK` | road 적용 뒤 crossing 좌표와 class≥2/`ROAD` 조건의 교집합 | 입구 회랑이 `ROAD`로 덮을 수 있고, 고립된 작은 보행 성분 collapse가 `BUILDING`으로 바꿀 수 있음 |

### 1.4 리옹 후처리: source terrain → 최종 terrain

리옹 전용 generator의 변형 순서는
[`build-lyon-city-geo.mjs` 451~487행](../scripts/build-lyon-city-geo.mjs#L451)에
드러난다.

1. `buildTerrainFromSnapshot()`으로 source terrain 생성.
2. `normalizeFrenchBridgeTerrain()`:

   - BRIDGE 성분이 서로 분리된 육지 접촉 2개 이상 또는 도로 접촉을 가지면 전부 `ROAD`.
   - 아니면 인접 수계에 따라 `RIVER`, 그 밖에는 `WATER`.
   - 최종 BRIDGE 0이 아니면 실패
     ([`build-french-city-geo.mjs` 158~227행](../scripts/build-french-city-geo.mjs#L158)).

3. POI·역을 투영하고 역을 반경 4타일 안 철도 mask에 스냅. 전 마커 간 Chebyshev
   이격 3타일을 검사.
4. 차단된 마커 셀을 `PLAZA`로 열고 4방 이웃이 모두 막혔으면 이웃 `PLAZA`도 연다.
5. Part-Dieu 입구부터 EXIT 앞까지 세로 회랑을 `ROAD`로 만든다. 수면을 만나면 실패.
6. Part-Dieu 기준 4방 BFS 정규화:

   - 40타일 미만이고 보호 마커가 없는 고립 보행 성분은 `BUILDING`으로 collapse.
   - 큰 성분은 수면을 건드리지 않는 최단 탐색으로 주 성분에 `ROAD`를 carve.
   - 보호 마커 성분이 수면 때문에 연결 불가하면 실패.
   - 최종 모든 보행 타일과 보호 마커가 단일 4방 성분인지 검사.

7. 마커 anchor를 한 번 더 보행 가능하게 보정.
8. 최종 건물·수계·연결성 통계를 meta에 기록하고 terrain/railway를 RLE로 직렬화해
   `lyon.geo.js`를 쓴다
   ([540~585행](../scripts/build-lyon-city-geo.mjs#L540)).

현 base의 실제 단계별 변화는 다음과 같다.

| 단계 | ROAD | SIDEWALK | CROSSWALK | BUILDING | BRIDGE | WATER | RIVER |
|---|---:|---:|---:|---:|---:|---:|---:|
| source terrain | 82,045 | 70,787 | 8,315 | 26,986 | 739 | 8,204 | 4,808 |
| 교량 정규화 후 | 82,765 | 70,787 | 8,315 | 26,986 | 0 | 8,206 | 4,825 |
| 최종 terrain | 82,562 | 68,586 | 8,311 | 29,414 | 0 | 8,206 | 4,825 |

교량 정규화는 `BRIDGE→ROAD 720`, `BRIDGE→RIVER 17`, `BRIDGE→WATER 2`다.
그 뒤 연결성 후처리는 대표적으로 `ROAD→BUILDING 209`,
`SIDEWALK→BUILDING 2,199`, `CROSSWALK→BUILDING 4`, `MOUNTAIN→ROAD 2`,
`BUILDING→PLAZA 4`를 만든다. 그러므로 Q1이 최종 geo에서 찾은 좌표를
`applySnapshotMasks()` 중간에 바로 적용하면 기존 후처리가 정정을 다시 덮을 수 있다.

### 1.5 “방향”은 별도 geo 필드가 아니다

26도시가 사용하는 `autotile-v1`에서 도로·횡단보도·교량 방향은 타일에 저장되지 않는다.
`CityScene.roadMask()`가 최종 terrain의 N/E/S/W road-like 이웃으로 4비트 mask를 만들고,
`mask=15`와 대각 네 칸이 road-like이면 `inner`를 고른다
([`CityScene.js` 1134~1168행](../src/components/world/CityScene.js#L1134)).

따라서 Q1의 “방향 어긋남”이 이웃 위상 오류라면 주변 타일 정정으로 해결할 수 있지만,
광폭 교차로처럼 같은 최종 mask에서 원래 way bearing을 잃은 경우에는 tile code
치환만으로 올바른 축을 표현할 수 없다. 기존 조사도 `mask=15`/`inner`에서 횡단보도
가로·세로가 동률이면 원래 축을 복원할 수 없다고 확인했다
([`proposal-autotile-expansion.md` 10~17행](proposal-autotile-expansion.md#L10),
 [131~135행](proposal-autotile-expansion.md#L131)).

Q2 후속 구현은 Q1 항목을 다음처럼 나눠야 한다.

- `tile-rewrite`: 정확한 `from→to` 치환으로 해결 가능한 고립 건물·부적절한 횡단보도·끊김.
- `topology-rewrite`: 주변 여러 셀의 원자적 치환으로 올바른 4방 mask를 복원할 수 있는 방향 오류.
- `needs-bearing-hint`: 같은 terrain mask로 축이 모호한 방향 오류. geo를 왜곡해 축을 강제하지
  말고 별도 source-bearing/axis 계약의 후속 SPEC로 넘긴다.

## 2. 정정 삽입 지점 두 안 비교

### 2.1 (a) 파이프라인 내 정정 패스 + geo 재생성

권장 구조는 공용 순수 함수와 도시별 고정 manifest다.

```text
pinned OSM snapshot
  → 기존 raster/classification
  → 기존 도시별 mutating postprocess
  → applyTileFixes(finalTerrain, cityFixManifest)
  → read-only final invariant validation
  → final stats/meta + canonical RLE serialization
  → committed {city}.geo.js
```

정정 manifest의 최소 계약은 다음과 같다.

```json
{
  "version": 1,
  "scannerContract": "tile-integrity-v1",
  "city": "lyon",
  "baseTerrainSha256": "<정정 전 final terrain SHA-256>",
  "fixes": [
    {
      "findingId": "<Q1 stable id>",
      "kind": "isolated-building",
      "x": 0,
      "y": 0,
      "from": 9,
      "to": 1,
      "contextSha256": "<고정 반경 neighborhood SHA-256>"
    }
  ],
  "expectedTerrainSha256": "<정정 후 SHA-256>",
  "expectedPngSha256": "<정정 후 render SHA-256>"
}
```

규칙:

- manifest 파일 자체도 SHA-256으로 고정하고 generator meta에
  `tileFixes: { version, manifestSha256, count }`를 남긴다.
- fix는 `y,x,findingId` canonical 순서로 적용한다. 중복 좌표·범위 밖·현재 값이 `from`과
  다름·context SHA 불일치는 즉시 실패한다.
- 한 finding이 여러 셀을 고치면 한 원자적 group으로 검증한 뒤 일괄 적용한다.
- 적용 뒤 자동 “수선”을 다시 돌리지 않는다. BFS·마커·수계·BRIDGE·도시 프로필을
  읽기 전용으로 검사하고 실패하면 manifest를 재검토한다.
- 정정 전 `baseTerrainSha256`는 scanner가 본 exact terrain을 고정해, upstream generator
  변경 뒤 낡은 정정이 조용히 다른 좌표에 적용되는 것을 막는다.
- `expectedTerrainSha256`와 `expectedPngSha256`는 의도된 핀 갱신이다. 스냅샷 SHA와
  raw Overpass SHA는 바뀌지 않는다.

리옹에서는 개념적으로 현재 마지막 `ensureWalkableAnchor()` 뒤
([`build-lyon-city-geo.mjs` 484~486행](../scripts/build-lyon-city-geo.mjs#L484)),
`finalBuildingStats` 계산 전이 hook이다. 다만 `connectivity.walkableTileCount`처럼 이미
계산된 파생 meta는 정정 뒤 읽기 전용 BFS 결과로 다시 확정하도록 finalizer를 나눠야 한다.
그렇지 않으면 terrain은 바뀌었는데 meta가 정정 전 수치를 가리키게 된다.

장점:

- 스냅샷 → generator → override → geo의 단일 provenance가 유지된다.
- clean input에서 두 번 실행하면 같은 geo bytes와 PNG bytes가 나오는 기존 계약을
  그대로 확장할 수 있다.
- generator 재실행이 정정을 지우지 않는다.
- `from`, base terrain SHA, context SHA 덕분에 upstream drift를 fail-closed로 잡는다.
- 도시별 통계·RLE·meta가 정정 후 terrain에서 함께 재계산된다.

비용:

- 현재 26개 산출물은 21개 geo generator 진입점에 걸쳐 있어 공용 hook을 연결하는
  초기 작업이 필요하다.
- exact count·terrain SHA·PNG SHA·RLE run 수를 pin한 도시별 테스트는 의도적으로
  갱신해야 한다.
- 정정으로 건물 비율·수계 단면·BFS가 바뀌면 단순 snapshot update가 아니라
  프로필 재실측과 Claude 검수 게이트가 필요하다.

### 2.2 (b) 산출 geo 직접 후처리 스크립트

가능한 형태는 커밋된 `*.geo.js`를 import/decode하고 tile을 고친 뒤 다시 RLE로 쓰는
스크립트다. 하지만 이 방식에는 세 가지 입력 정의가 생긴다.

1. 커밋된 geo를 입력으로 삼으면 generator를 다시 돌렸을 때 정정이 사라진다.
2. 매번 generator로 임시 base geo를 만든 뒤 후처리하면 실질적으로 (a)의 외부 wrapper이며,
   정정 패스가 파이프라인 일부라는 사실만 숨긴다.
3. 이미 후처리된 geo에 in-place로 다시 적용하면 `from`/base hash 계약과 재실행
   idempotence가 충돌한다.

추가 충돌:

- 26개 geo의 최상위 payload가 동일하지 않다. `transitPoints`, `tileSkins`, `tide`,
  `entrance/exitTiles`, `railways` 유무가 달라 공용 재직렬화기가 필드를 누락하거나
  표현을 바꿀 위험이 있다.
- 모든 파일의 헤더는 generator 단독 산출이라고 선언한다. 후처리 provenance를 별도로
  기록하지 않으면 주석이 거짓이 된다.
- 도쿄·오사카·교토 등 테스트는 `build*Geo()` terrain과 커밋된 geo terrain의 SHA가
  같음을 직접 검사한다. 후처리만 하면 즉시 불일치한다.
- 리옹 테스트도 고정 terrain/PNG SHA, RLE run 수, 지형별 exact count,
  수계·연결성 meta를 고정한다
  ([`lyonGeo.test.js` 276~327행](../src/components/world/__tests__/lyonGeo.test.js#L276)).
- 후처리 스크립트가 최종 파일만 수정하면 generator 내부의 건물·연결성 meta는
  정정 전 값을 유지한다.

장점은 공통 모듈 import만 가능하면 26개 파일에 한 번에 접근하기 쉽고, generator가
없는 legacy 파일도 임시로 다룰 수 있다는 점이다. 그러나 현 저장소에는 그런 legacy
파일이 없으므로 이 장점은 현재 문제에 적용되지 않는다.

### 2.3 트레이드오프 요약

| 기준 | (a) pipeline pass | (b) output postprocessor |
|---|---|---|
| 단일 진실원 | 스냅샷+generator+manifest 한 경로 | generator 결과와 커밋 geo가 분리 |
| clean 2회 byte-identical | 자연스럽게 충족 | fresh base를 별도로 만들지 않으면 입력 정의가 흔들림 |
| upstream drift | base terrain/context SHA로 fail-closed | 커밋 파일 hash는 잡아도 generator drift와 분리 |
| 기존 hash pin | 의도적 terrain/PNG/count 핀 갱신 | generator-vs-committed 비교를 깨뜨림 |
| final meta | 정정 후 재계산 가능 | 별도 범용 serializer/finalizer 필요 |
| `verify-city-geo.mjs` | 정정 후 최종 파일 검증 | 최종 파일은 검증 가능하지만 provenance 불일치는 못 잡음 |
| 26도시 확산 | 21 generator hook 연결 비용 | 한 스크립트처럼 보이나 payload 6종 이상·serializer 차이 |
| 재현 가능성 | 높음 | committed geo를 암묵적 source로 삼으면 낮음 |
| legacy 무소스 대응 | 먼저 generator 이관 필요 | 임시 대응 가능 |
| 현재 legacy 필요성 | 없음 | 없음 |

## 3. 결정성·검증 계약과 충돌 지점

### 3.1 현재 리옹 핀

[`lyonGeo.test.js`](../src/components/world/__tests__/lyonGeo.test.js)는 다음을 동시에
고정한다.

- 스냅샷 파일 SHA와 raw Overpass SHA.
- bbox/grid/투영/언어 schema.
- POI·역 좌표와 ≤2.5타일 재투영, 마커 이격.
- 수계 final tile count·단면, BRIDGE=0.
- 전체 보행 타일 단일 4방 BFS.
- 지형별 exact tile count.
- 두 번 `buildLyonCityGeo()` terrain SHA 동일.
- terrain SHA `1d3036...`, railway SHA `f47e8c...`.
- terrain RLE 93,064 runs, railway RLE 8,089 runs.
- 두 번 렌더 PNG byte-identical과 PNG SHA `2d1aac...`.

Q2 정정이 한 셀만 바꿔도 terrain SHA와 대개 PNG SHA·지형 count가 바뀌며, run 경계에
따라 RLE run 수도 바뀐다. 이는 결정성 위반이 아니라 **의도된 새 canonical output**이다.
다만 같은 manifest로 A/B clean run이 같고, 변경된 핀이 Q1 finding과 1:1로 설명돼야 한다.

### 3.2 공용 verifier의 범위

리옹 프로필은
[`verify-city-geo.mjs` 433~451행](../scripts/verify-city-geo.mjs#L433)에서 다음을
hard gate로 둔다.

- `20m/tile`, `contentLocale=fr`
- 건물 13~17%
- POI 재투영 ≤2.5타일
- 론강 단면 합계/연속 ≥140m
- 손강 단면 합계/연속 ≥50m
- BRIDGE 잔존 0

공용 verifier는 최종 terrain을 읽어 이 의미 계약을 검사하지만
([624~715행](../scripts/verify-city-geo.mjs#L624)), geo 파일 전체 SHA나 terrain SHA,
두 번 재생성을 검사하지 않는다. 따라서:

- (a)는 도시별 determinism test와 A/B 생성으로 byte 계약을, verifier로 의미 계약을
  각각 충족해야 한다.
- (b)는 verifier가 green이어도 generator-vs-committed 불일치와 provenance 분리를
  발견하지 못한다.
- 26도시 일괄 게이트는 `verify-city-geo.mjs`만 반복해서 끝나지 않는다. 프로필이 없는
  도쿄·오사카·교토·후쿠오카 4개는 각 geo Vitest의 generator equality·BFS·질량 계약을
  반드시 함께 실행해야 한다.

### 3.3 Q2 후속 구현의 필수 증거

도시별:

1. Q1 finding ID → manifest fix group → 변경 좌표 목록의 완전 대응.
2. 정정 전 terrain SHA, manifest SHA, 정정 후 terrain SHA.
3. clean base에서 독립 A/B 생성한 geo 파일 byte SHA 일치.
4. 독립 A/B PNG SHA 일치.
5. scanner 재실행 시 승인 finding 0, 새로운 4유형 finding 0.
6. 도시별 geo 테스트와, 지원 도시라면 `verify-city-geo.mjs --city <id>` green.
7. 4방 BFS, 마커 ≥3타일, POI ≤2.5타일, BRIDGE 정책, 미니맵 <24MiB 불변.

전 도시 batch:

1. 도시 ID를 manifest canonical 순서로 고정하고 26/26 입력/출력 SHA 표를 만든다.
2. 정정이 없는 도시도 generator output이 committed geo와 같은지 검사한다.
3. 22개 공용 verifier profile + 일본 4도시 geo Vitest를 합쳐 26/26 의미 게이트를 만든다.
4. full pipefail Vitest를 마지막에 실행한다.

### 4.2 미래 legacy 예외 정책

향후 OSM 스냅샷·generator가 없는 geo가 추가로 발견되면 기본 경로는 다음과 같다.

1. 기존 geo와 라이선스가 확인되는 원본을 보존한다.
2. source identity·license·SHA·bbox·projection을 pin한 최소 generator로 먼저 이관한다.
3. 이관한 generator의 출력이 기존 geo와 byte-identical인지 증명한다.
4. 그 뒤 (a)의 manifest pass를 적용한다.

원본을 법적·기술적으로 복구할 수 없는 경우에만 (b)를 격리된 임시 migration으로 허용한다.
이때도 입력 geo SHA, 후처리 script SHA, fix manifest SHA, 출력 geo/PNG SHA를 모두 pin하고,
생성 헤더에 두 단계 provenance를 기록해야 한다. 이 예외를 26도시 기본 경로로 승격하면
안 된다.

## 5. 권장 적용 순서

1. Q1이 stable finding ID, exact city/x/y, 관찰한 tile code, neighborhood hash,
   scanner version을 내보내게 한다.
2. finding을 `tile-rewrite`, `topology-rewrite`, `needs-bearing-hint`로 수동 분류한다.
3. 우선 리옹 한 도시에서 공용 순수 `applyTileFixes()`와 manifest schema를 검증한다.
4. 리옹 generator의 마지막 mutating 후처리와 final meta/RLE 사이에 hook을 둔다.
5. A/B geo·PNG, Q1 rescan, Lyon geo test, Lyon verifier, full Vitest를 통과시킨다.
6. generator 계열 단위로 hook을 확산한다. 공용 generator부터 적용하고 전용 generator는
   같은 conformance test로 묶는다.
7. 26/26가 같은 manifest contract와 batch gate를 통과한 뒤에만 전면 정정을 선언한다.

최종 판단은 **(a) GO / (b) NO-GO(legacy emergency only)**다. 현재는 모든 도시가
재생성 가능하므로 provenance·결정성·확산성을 동시에 만족하는 유일한 기본 경로가 (a)다.

## 6. Q3 구현: 리옹 no-op hook과 manifest v1

Q3는 권장 순서 3~4단계의 골격만 구현한다. 공용 순수 함수
[`applyTileFixes()`](../scripts/lib/applyTileFixes.mjs)를 만들고, 리옹 generator의 마지막
`ensureWalkableAnchor()` 뒤이자 `finalBuildingStats`·hydrology count·RLE 산출 전인
[`build-lyon-city-geo.mjs`](../scripts/build-lyon-city-geo.mjs)에 연결했다. 아직 승인된
정정 좌표가 없으므로 리옹 manifest의 `fixes`는 비어 있으며, geo meta나 생성 파일에
빈 manifest 정보를 추가하지 않는다. 따라서 기존 geo bytes와 hash pin은 바뀌지 않는다.

### 6.1 manifest v1 스키마

도시 manifest의 공통 envelope는 다음과 같다.

| 필드 | 계약 |
|---|---|
| `version` | 현재 `1`; 다른 버전은 fail-closed |
| `city` | manifest가 적용되는 canonical city ID |
| `grid.w`, `grid.h` | flat row-major grid의 폭·높이; `w*h === grid.length` |
| `scannerVersion` | finding을 만든 scanner 계약 버전 |
| `fixes` | 정정 배열; 빈 배열은 명시적 no-op |

각 `fixes[]` 항목은 `findingId`, `city`, `x`, `y`, `before`, `after`, `ruleId`,
`scannerVersion`을 필수로 가진다. `city`와 `scannerVersion`은 envelope와 같아야 하고,
좌표는 격자 안의 정수, tile code는 `0..255` 정수여야 한다. `before === after`,
중복 좌표, 현재 grid 값과 `before`의 불일치는 모두 실패한다. Q2의 설명용
`from`/`to` 표기는 Q3 구현 스키마에서 Q1 scanner 용어와 맞춘 `before`/`after`로
확정한다.

### 6.2 순수 함수와 hook 계약

- `applyTileFixes(grid, manifest) => grid`는 입력 grid와 manifest를 수정하지 않는다.
- 빈 `fixes`는 검증 후 입력 grid의 같은 객체를 그대로 반환한다.
- non-empty fix는 `y`, `x`, `findingId` canonical 순서로 정렬하되 manifest 배열을
  재배열하지 않는다.
- 모든 항목의 스키마·좌표·중복·`before`를 먼저 검증한 다음 복사한 grid에 적용한다.
  따라서 한 항목이라도 실패하면 부분 적용된 결과가 없다.
- hook 뒤에는 지형을 바꾸는 패스를 두지 않는다. final 통계와 RLE는 hook 반환값만
  소비한다.
- Q3는 no-op이므로 기존 connectivity meta가 정확히 유지된다. 최초 non-empty
  manifest를 넣는 후속 단계에서는 hook 뒤 4방 BFS·마커·수계·BRIDGE 검사를 읽기
  전용으로 다시 수행하고, 파생 meta와 terrain/PNG hash pin을 정정 결과로 갱신해야 한다.

리옹의 Q3 빈 manifest는 `version=1`, `city=lyon`, `grid=428×501`,
`scannerVersion=tile-integrity-v1`, `fixes=[]`로 고정한다. helper 단위 테스트는
빈 manifest의 reference identity, 입력 불변, canonical 적용, 전항목 선검증,
중복 좌표와 스키마 drift의 fail-closed 동작을 고정한다.

### 6.3 no-op 산출 증거

공식 Node v22.23.1에서 hook 삽입 전후 각각 두 번 독립 재생성한
`src/components/world/cities/lyon.geo.js`의 SHA-256은 다음과 같다.

| 시점 | A | B |
|---|---|---|
| hook 삽입 전 | `0bb8a0a2983df4cfbaa2387fa8905610508ef5ecaa41d5d3992bf33ca41936d2` | `0bb8a0a2983df4cfbaa2387fa8905610508ef5ecaa41d5d3992bf33ca41936d2` |
| hook 삽입 후 | `0bb8a0a2983df4cfbaa2387fa8905610508ef5ecaa41d5d3992bf33ca41936d2` | `0bb8a0a2983df4cfbaa2387fa8905610508ef5ecaa41d5d3992bf33ca41936d2` |

네 파일은 byte-for-byte 동일하고 커밋된 `lyon.geo.js`와도 같으므로 생성 산출물과
기존 snapshot/hash pin 변경은 없다.
