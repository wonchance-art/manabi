# 7도시 개방 지구 콘텐츠 갭 실측

## 결론

`district-v1`이 적용된 7도시 26개 개방 지구를 정본 마스크 그대로 재어 보니, 개방 영역 안에는
고유 기준으로 도어 3개·NPC 0개·발견 스팟 8개·POI 87개·TRANSIT 도착 35개가 있다. 그러나
**보르도 `gare-saint-jean`과 스트라스부르 `gare`는 고유 콘텐츠 앵커가 0개**다. 두 지구 모두
도시 입구와 표시용 역은 있지만, 역이 `transit[].stopIds`에 소비되지 않아 TRANSIT 도착으로 세지
않는다.

시스템 분포도 한쪽으로 기울어 있다. 도어 3개는 모두 레만호 연안, 발견 8개는 모두 리옹에 있고,
NPC는 26개 지구 전부 0개다. 따라서 다음 채움 라운드는 숫자가 많은 POI를 더 늘리기보다
**빈 역 지구 → 저밀도 광역 지구 → 언어 상호작용이 없는 POI 전용 지구** 순으로 도어·NPC·발견을
분산하는 편이 낫다.

## 측정 기준과 재현성

- 정본: `origin/main` `d3d69f07c0d1be0046fc3533cb61c684fa9b1903`, Node `v22.23.1`.
- 대상·순서: `lyon`, `bordeaux`, `strasbourg`, `seoul`, `busan`, `cote-dazur`,
  `leman-riviera`; 각 도시의 `districts.version === 'district-v1'`과 `districts.open` 배열을 그대로
  소비했다.
- 면적: 지구별 rect 합집합의 고유 타일 수 × `20m × 20m`. 기존 콘텐츠 밀도 감사처럼 수면·차단
  타일도 포함한다. 이는 지구 마스크 안에서 플레이어가 체감하는 전체 공간 공백을 보는 지표이며
  보행 가능 토지의 품질 점수가 아니다.
- 도어: runtime `city.nodes` 중 `chapter`가 문자열인 노드. NPC: `kind === 'npc'` 또는
  `npc === true`인 노드.
- 발견: `resolveCityMainRoute()`가 확정한 발견 타일. POI: 도시 geo 정본 `GEO.pois`의 id와 일치하는
  runtime 노드만 센다. 따라서 레만호의 `fr-16`~`fr-18` 도어를 POI에 중복 산입하지 않는다.
- TRANSIT 도착: 모든 `city.transit[].stopIds`의 고유 id를 exact-1 역/도선 포인트에 연결한 뒤
  `resolveArrivalTile()`이 반환한 실제 하차 타일로 센다. 표시 전용 역은 제외한다. 7도시에서 원본
  정차 타일과 실제 하차 타일이 달라진 사례는 0개다.
- 밀도: 위 다섯 범주의 **서로 다른 실제 타일 수 / 지구 마스크 km²**. 한 타일의 도어·POI·도착이
  겹치면 고유 앵커 1개로 센다.
- 지구 마스크끼리 겹치면 해당 타일은 포함하는 모든 지구에 귀속한다. 이 때문에 지구 행의 합은
  도시 고유 합보다 클 수 있으며, 중첩은 아래에 별도 공개한다.
- 동일 one-shot Node 측정기 2회 JSON은 각 33,036 bytes로 byte-identical. SHA-256
  `139b9241bca67790fa281eb67a86ac6f5676dcdf6b12d19f4e185ea186f54abb`.
- 전수 측정기: 0.95초, max RSS 789,495,808 bytes, peak footprint 793,802,624 bytes, swap 0.
- targeted 지구·레지스트리·발견 회귀: 4 files / 54 tests PASS(13.78초), max RSS
  2,324,070,400 bytes, peak footprint 2,926,241,296 bytes, swap 0.
- 전체 회귀: 211 files / 2,137 tests PASS(183.26초), max RSS 3,430,383,616 bytes,
  peak footprint 3,339,704,872 bytes, swap 0. `npm run lint`와 `git diff --check`도 PASS.

## 도시 고유 합과 잠금 영역

아래 `개방`은 지구 중첩을 제거한 도시별 고유 항목 수다. `잠금 POI`는 어떤 개방 마스크에도 들지
않는 geo POI이며, 이번 채움 대상에 섞지 않는다.

| 도시 | 개방 도어 | 개방 NPC | 개방 발견 | 개방 POI | 개방 T도착 | 잠금 POI |
|---|---:|---:|---:|---:|---:|---:|
| 리옹 | 0 | 0 | 8 | 7 | 2 | 2 (`confluence`, `tete-dor`) |
| 보르도 | 0 | 0 | 0 | 9 | 0 | 0 |
| 스트라스부르 | 0 | 0 | 0 | 7 | 0 | 0 |
| 서울 | 0 | 0 | 0 | 20 | 8 | 3 (`amsa-dong`, `gimpo-airport`, `seoul-nat-univ`) |
| 부산 | 0 | 0 | 0 | 14 | 7 | 0 |
| 코트다쥐르 | 0 | 0 | 0 | 18 | 6 | 0 |
| 레만호 연안 | 3 | 0 | 0 | 12 | 12 | 0 |

리옹만 open rect가 의도적으로 겹친다. 중첩 항목은 발견 `lyon-d5`·`lyon-d6`, POI
`croix-rousse`·`opera`·`terreaux`·`vieux-lyon` 6개다. 다른 6도시에는 콘텐츠 항목의 지구
중첩이 없다. 이 공개 규칙 덕분에 리옹 지구 행을 단순 합산해 도시 고유 수로 오해하지 않는다.

## 채움 우선순위 제안

### P0 — 콘텐츠 0인 두 역 지구

1. **보르도 `gare-saint-jean`**: 1.0004km²에 도어·NPC·발견·POI·TRANSIT 도착이 모두 0이다.
   표시용 `bordeaux-saint-jean` 역은 존재하지만 노선이 없어 플레이 장치가 아니다. 다음 라운드에서
   역 지구에 도어 1개와 재사용 가능한 역무 NPC 1개를 먼저 배치하고, TRANSIT은 노선 SPEC을 따로
   확정한다.
2. **스트라스부르 `gare`**: 0.2992km²가 동일하게 전부 0이다. 중앙역을 POI 숫자에 편입해 수치만
   올리지 말고, 도어 또는 역무 NPC와 짧은 발견 1개를 결합해 진입 직후 상호작용을 만든다.

### P1 — 0.25 앵커/km² 미만의 광역 공백

| 대상 | 현재 증거 | 다음 라운드 최소 제안 |
|---|---|---|
| 레만호 `lavaux` | 50.2784km², 0.159; 도어 1·도착 4가 있으나 장거리 회랑 | 새 POI 대신 기존 역 사이에 발견 2개와 역할 NPC 1개를 분산 |
| 서울 `river-north` | 11.0440km², 0.181; POI 2뿐, 도어·NPC·발견·도착 0 | 기존 POI 중 하나에 학습 도어, 서로 다른 타일에 NPC/발견 각 1개 |
| 서울 `west` | 16.3684km², 0.183; POI 2·도착 1뿐 | 도어 1·NPC 1·발견 1을 한 지점에 몰지 않고 서남권 안에 분산 |
| 코트다쥐르 `nice` | 43.6644km², 0.183; POI 7·도착 1이지만 언어 장치 0 | 불어 도어 1·역할 NPC 1·해안/구시가 발견 2개를 기존 앵커에 연결 |
| 코트다쥐르 `est` | 18.7264km², 0.214; POI 2·도착 2, 언어 장치 0 | 발견 1·NPC 1을 서로 다른 정차권에 배치; 도어는 니스와 세트 비중복 검토 |

### P1 — POI만 있고 플레이 장치가 없는 보르도·스트라스부르 본체

- 보르도 `centre-historique`·`nord-rive`, 스트라스부르 `grande-ile`·`quartier-europeen`은
  공간 밀도 자체는 1.850~3.754로 높지만 콘텐츠가 POI 한 종류뿐이다. 두 도시 모두 도어·NPC·발견·
  TRANSIT 도착이 도시 전체에서 0이므로, 역 지구 P0 뒤에 **도시당 도어 세트 1개와 NPC 1개**를
  POI가 많은 중심 지구와 외곽 지구로 나눠 넣는다.
- 발견은 POI 설명을 복제하지 않고 역~중심가의 이동 구간에 둔다. 정확 타일·카피·노선은 Claude의
  후속 콘텐츠 SPEC에서 확정한다.

### P2 — 한국 해안·강남과 레만 동단의 다양성 보강

- 부산 `east-coast`(0.289)와 `port-core`(0.469)는 도착이 있어 이동은 되지만 도어·NPC·발견이
  모두 0이다. 도시 전체에 학습 상호작용이 없으므로 두 지구를 첫 세트로 묶고 도어 1개씩보다
  **도어 1개 + NPC/발견 분산**을 우선한다.
- 부산 `south-coast`는 TRANSIT 도착 0이지만 POI 4개가 흩어져 있다. 새 역을 숫자 맞추기로 만들지
  말고 기존 POI 사이의 발견 또는 현장 역할 NPC를 먼저 검토한다.
- 서울 `southeast`(0.485), 레만 `montreux-chillon`(0.604)은 이동 앵커는 충분하다. 새 POI보다
  각각 도어/NPC, NPC/발견처럼 빠진 장치 종류를 보강한다.

### P3 — 현재 밀도 유지·과채움 방지

- 리옹 4지구는 2.991~6.374 앵커/km²이고 발견 8개가 이미 전 주동선을 잇는다. 도어·NPC 공백은
  남지만 발견 수를 더 늘리지 않는다. 후속 불어 도어 1개와 역할 NPC 1개를 서로 다른 지구에 두는
  정도가 적절하다.
- 코트다쥐르 `ouest`·`monaco`, 레만 `lausanne-ouchy`·`vevey`, 부산 `central-north`, 서울
  `historic-core`는 비교적 앵커가 충분하다. P0/P1 지구보다 먼저 숫자를 맞추지 않는다.

## 비구현 경계

이 문서는 현재 정본의 실측과 채움 순서만 제안한다. 도어 카피·NPC 역할/대사·발견 문장·TRANSIT
노선·지구 rect·geo·runtime·registry·verifier·DB 변경은 승인하지 않는다. 잠금 POI를 개방 지구로
옮기거나 rect를 확장하는 것도 별도 SPEC이 필요하다.
